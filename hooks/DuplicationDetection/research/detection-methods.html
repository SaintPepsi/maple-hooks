01 — Exact Match

Structural Hash

Two functions are literally the same code, copy-pasted. Different files, different variable names, but the structure is identical.

#

How It Works

Baseline
Exact Match

Take a function's AST. Strip all identifiers and literal values — variable names, strings, numbers. Keep only node types and structure. Serialize the normalized AST to a canonical string. SHA-256 hash it.

Two functions with the same hash have the same skeleton, even if every variable name differs.

// Before normalization const result = deps.readFile(path); if (!result.ok) return result; // After normalization (identifiers + values stripped) const _ = _._(_); if (!_._) return _; // Both produce the same hash: aeb7b87c0ce63c80

✓ What It Catches

makeSourceRepo in 4 test files. runHook in 11 test files. execute in 4 fixture files. All structurally identical.

✗ What It Misses

Functions doing the same thing with one extra if-branch, different parameter order, or any structural variation at all.

Metric Value
Clusters found 47
Precision 100% (zero false positives)
Top finding runHook — 11 identical copies across test files
02 — Approximate Match

N-gram Subsequence

Functions follow the same template but aren't exactly identical. They share characteristic patterns like "declare variable, call method, check result, return."

N

How It Works

Cycle 1
Approximate Match

Flatten each function's AST into a sequence of node types. Generate sliding windows of N consecutive types (n-grams). Score function pairs using Jaccard similarity: shared n-grams divided by total n-grams. Cluster above threshold.

// Function body as node type sequence [BlockStatement, VariableDeclaration, CallExpression, IfStatement, ReturnStatement, ...] // 4-grams extracted BlockStatement→VariableDeclaration→CallExpression→IfStatement VariableDeclaration→CallExpression→IfStatement→ReturnStatement ...

Two functions sharing many 4-grams follow the same template, even if their overall structure differs.

✓ What It Catches

getFilePath + getCommand + getWriteContent across 17 hook contracts. Same "check input, extract field, return" subsequence.

✗ What It Misses

Functions with the same intent but completely different control flow. Top n-grams are too generic — CallExpression→Identifier appears everywhere.

03 — Approximate Match

CFG Skeleton SUPERSEDED

Functions share the same branching logic regardless of what's inside the branches. Not used in production. The body fingerprint (used in the composite ranker and production hooks) subsumes this signal with better granularity — it captures control flow node frequencies as part of a 16-element AST frequency vector with continuous cosine similarity, rather than binary skeleton matching.

◊

How It Works

Cycle 2 — Research Only
Approximate MatchNot In Production

Extract only control flow nodes from the body: IfStatement, ForStatement, ReturnStatement, TryStatement, CatchClause. Track nesting depth. The result is a "skeleton" — the pure branching structure with all expression-level detail stripped.

// Safe wrapper skeleton (10 functions share this) TryStatement → ReturnStatement → CatchClause → ReturnStatement // Read-validate-return skeleton (7 functions) TryStatement → ReturnStatement → CatchClause → IfStatement → ReturnStatement(2)

Four fingerprint variants tested: full (with depth), compressed (depth-free), shape (unordered), abbreviated (collapsed runs).

Why superseded: Too coarse at the top end (77 functions match If→Return), too sparse at the precise end (12 clusters at depth 1). The body fingerprint already captures control flow structure as part of its frequency vector, making this a redundant signal. Its contribution was as a research stepping stone toward the fingerprint design.

✓ What It Catches

The "safe wrapper" pattern: 10 functions across adapters (safeRegexTest, createRegex, safeParseYaml, tryCatch) all share try→return→catch→return.

✗ What It Misses

Functions where control flow differs even slightly. Very high precision but low recall — only 12 clusters at strict settings.

04 — Semantic Match

Role Naming

44 functions named makeDeps across 44 files. Each builds a different Deps object, so structural analysis can't group them. The signal is in the name.

Aa

How It Works

Cycle 3
Semantic Match

Group functions by exact name across files. Validate structurally — compute average pairwise body similarity. If >30%, it's a genuine pattern. Also normalizes names to extract "roles" (makeDeps → role deps) to catch variants like ensureDir + ensureDirSafe.

Classifies architectural verb: factory (make/create/build), accessor (get/read), predicate (is/has), formatter, parser, handler, validator.

File Template

06 — Approximate Match

Type Signature

57 adapter functions share the type signature (string) → Result and have 74% body similarity. Different names, different imports — the only consistent signal is the types.

T

How It Works

Cycle 5
Approximate Match

Extract type signature per function: (paramType1, paramType2) → returnType. Group by exact signature. Within each group, compute body similarity using condensed fingerprints — cosine similarity on a 16-element frequency vector of the most common AST node types, encoded as a 32-char hex string.

Flag "name-diverse" clusters (2+ distinct function names) as the novel findings — these are invisible to all name-based approaches.

💡 Highest Coverage Detector

708 functions in name-diverse clusters — 85% of the codebase. 98% of clusters have 2+ distinct names, proving these patterns are genuinely invisible to name-based detection. The type signature pre-filter keeps it cheap: 43ms total for all 837 functions.

07 — Fusion ⭐ THE PRODUCTION ENGINE

Composite Ranker POWERS THE HOOK

This is the core of the production DuplicationChecker hook. Every other detector was research leading to this. Five detectors, five separate outputs — the composite ranker fuses them into one answer: "what should I refactor first?" Its dimension thresholds (4/4 = block, 3/4 = suggest, <3 = silent) are the exact logic running in real-time on every Write/Edit.

⭐

How It Works

Cycle 6 — Production
Multi-Signal FusionProduction Hook Logic

Scores every function across 4 dimensions: structural hash (weight 1.0), body similarity (0.8), name repetition (0.7), type signature commonality (0.4). Counts how many dimensions fire. Groups by function name, ranks by average dimension count then score.

A function hitting 4/4 dimensions is near-certainly duplicated. The dimension bar shows confidence at a glance:

●●●● runHook — 13 instances, all signals at 100% ●●●● makeInput — 31 instances, 3 signals ●●●● categorize — moderate signal, needs review ●●●● utils — weak signal, likely coincidental
Dimensions Functions Interpretation
●●●● 69 Near-certain refactoring targets
●●●● 145 Strong candidates
●●●● 387 Moderate signal
●●●● 132 Weak signal
●●●● 149 Genuinely unique

💡 Instance Count is Misleading

makeDeps has 44 instances but only 3 dimensions. runHook has 13 instances but all 4 dimensions at 100%. Dimensional convergence is a better proxy for "should definitely refactor" than raw count.

08 — Semantic Match

Co-occurrence Mining

Functions that always appear together reveal template patterns invisible to individual function analysis.

{}

How It Works

Cycle 8
Semantic Match

Frequent itemset mining (like market basket analysis). Count all function name pairs that co-occur in 3+ files. Expand pairs into larger tuples using Apriori: if {A, B} and {A, C} are both frequent, check if {A, B, C} is too. Filter to maximal tuples (remove subsets). Validate each tuple with body similarity.

Tuple Files Body Sim
